Architecture ‚Äî Three Components
  • Team Lead ‚Äî your main Claude Code session. Creates team, assigns tasks, synthesizes. In delegate mode: coordinates, does not implement.
  • Teammates ‚Äî separate Claude Code instances. Own context windows. Load same CLAUDE.md + MCPs. No inheritance of lead's conversation.
  • Shared task list ‚Äî markdown file. Agents claim tasks, update progress, auto-unblock dependencies. File locking prevents conflicts.
  • Mailbox system ‚Äî direct peer-to-peer messaging between any agents. No lead required as intermediary.
  • Key distinction: subagents report to one parent, can't talk to each other. Teammates message directly, challenge findings, self-coordinate.
  • Task states: pending ‚Üí in_progress ‚Üí completed. Teammates self-claim next unblocked task on completion.

Day 07
Spawn Prompts, Display Modes & Lifecycle
Phase 4 ‚Äî Agent Team Mechanics
~4 hrs ‚úì Done
Spawn Prompts ‚Äî The Most Important Skill
  • Teammates don't inherit context. A vague spawn prompt = a confused agent.
  • Every spawn prompt must include: exact files/scope, acceptance criteria, what to do on completion, who to report to
  • Team size sweet spot: 5‚Äì6 agents. Beyond that, diminishing returns set in fast.
Good spawn prompt 
Review app/models/user.rb for N+1 queries.
Check all ActiveRecord associations have proper includes.
Document findings in review-notes.md.
Send summary to team-lead via Teammate write.
Display Modes
  • In-process ‚Äî all teammates in main terminal. Fastest, no visibility into individual agents.
  • tmux split panes ‚Äî each teammate gets its own pane. Shift+Down to cycle. Recommended default.
  • iTerm2 ‚Äî teammates as split panes in your iTerm2 window. Set up today and leave as default.
Lifecycle Rules
  • Always shut down teammates before cleanup. The lead runs cleanup ‚Äî teammates' context may not resolve correctly.
  • Run cozempic checkpoint before any shutdown to preserve team state to disk.
  • Use Shift+Up/Down in in-process mode to select and message individual teammates directly.

Competing Hypotheses & Contract-First Spawning
Phase 4 ‚Äî Advanced Patterns
~5 hrs ‚úì Done
The Competing Hypotheses Pattern

The killer use case that single agents structurally can't replicate. Sequential investigation anchors on the first plausible theory. Multiple independent investigators trying to disprove each other converge on actual root causes significantly faster.

Debugging swarm prompt 
Production API returning 500s intermittently.
Create a debugging team with 3 agents:

Agent A: investigate database connection pool exhaustion
Agent B: investigate race condition in the caching layer
Agent C: investigate memory leak in the request handler

Share evidence via mailbox. Challenge each other's findings.
Report consensus (or dissent) to team-lead when complete.
Contract-First Spawning
  • Before spawning any agents to build together, have the lead create a contract.md ‚Äî endpoint specs, data shapes, error handling conventions.
  • Both frontend and backend agents read it before writing a line of code.
  • Prevents the most common failure: two agents building toward incompatible assumptions that only surface at integration time.

Day 09
Quality Gates, Hooks & Cost Management
Phase 4 ‚Äî Production Discipline
~4 hrs ‚úì Done
Team-Specific Hooks
  • TeammateIdle: runs when a teammate is about to go idle. Exit code 2 sends feedback and keeps them working. Use to enforce "don't stop until tests pass."
  • TaskCompleted: runs when a task is being marked complete. Exit code 2 prevents completion. Use to enforce peer review before task closure.
  • Exit code 2 pattern: hook outputs feedback message ‚Üí Claude reads it ‚Üí continues working. This is how you enforce standards without manual intervention.
Cost Management
  • Each teammate is a separate Claude instance ‚Äî costs ~5√ó a single session. Model mixing is not optional at scale.
  • Opus 4.6 for the lead/orchestrator (strategic decisions, synthesis, coordination).
  • Sonnet 4.6 for executor teammates (implementation, review, testing). ~3‚Äì4√ó cheaper.
  • Reserve teams for work where parallel perspectives genuinely add value. Single sessions for everything else.
Execution Patterns That Work at Scale
  • Feature squads ‚Äî each teammate owns a separate module, no shared files. Zero merge conflicts.
  • Research swarms ‚Äî 4‚Äì6 agents investigating different aspects of the same problem, sharing findings.
  • Builder-validator chains ‚Äî one agent builds, another immediately reviews and tests. Feedback loop without the lead intervening.
